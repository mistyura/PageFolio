<objective>
`OCRRunEngine`（03-01 で抽出）と `pagefolio/ocr_pipeline.py` を経由する OCR→サマリの一気通貫フローを、実 API 非依存の E2E モックテストとして `tests/test_ocr_engine.py` に整備する（V180-QA-01）。D-13 に従い実際に `threading.Thread` を起動し `queue.Queue` を通す高忠実度の統合テストとし、抽出後の `OCRRunEngine` コードパス（ワーカー起動・`PipelineState` 共有・sentinel 送出・最終ワーカー判定・完了理由別コールバック）を通す。D-14 に従い既存 `FakeProvider` パターンを複製・拡張して使い、E2E 専用の新規フェイク実装はしない。

Purpose: 単一ファイル OCR とバッチ OCR で共用される `OCRRunEngine` の中核コードパスを、タイミング依存の flaky を避けつつ実 API 非依存で保証する。これにより Phase 4（バッチ OCR）が同エンジンを安心して再利用できる回帰網を確立する。
Output: `tests/test_ocr_engine.py` の `TestOCRRunEngineE2E`（6シナリオ）+ サマリ生成用に拡張した `FakeProvider`。

D-15 のカバレッジ範囲（正常系 + 異常系 + サマリ生成）を狭めない。サマリ生成（`complete_text_ex` 相当の text-only 応答）は Pitfall 4/A2 に従い `OCRRunEngine` へ統合せず、「OCR 実行（Engine 経由）→ その後サマリ生成」というフロー全体をモックで検証する。
</objective>
