## Review Incorporation (Antigravity 04-REVIEWS.md)

本プランに対するクロスAIレビュー（Antigravity・04-REVIEWS.md）の actionable finding の反映状況。04-02 はリスク MEDIUM 評価で、以下の懸念を本プランのタスク/受入基準/threat_model/must_haves へ組み込んだ。

- **懸念1（HIGH）ダイアログクローズ時のスレッドリーク**: 反映済み。Task 2 で `self.protocol("WM_DELETE_WINDOW", self._on_close)` を `__init__`/`_build` で束ね、`_on_close` は実行中なら2階層フラグ（`_batch_cancel_flag`/`_file_cancel_flag`）を set → `self._run_gen += 1` で世代無効化 → `self.destroy()` の順に処理する（既存 `OCRDialog._on_close`（ocr_dialog.py:1959-1978）の踏襲）。threat_model に T-04-06 を追加。
- **懸念2（MEDIUM）`test_no_unused_lang_keys` 段階実装競合**: 反映済み。新規 lang キーは**消費するタスクと同一タスクで lang.py へ追加**する規約を明文化（`batch_scanning_msg` は Task 1・`batch_empty_queue_msg` は Task 2）。状態ラベルキー（`batch_status_*`）は Task 1 で `_STATUS_LABEL_KEYS`（status→lang キーの写像）としてソース内に文字列リテラルで出現させ、`_refresh_queue_row` が消費する。これにより各タスクの `pytest tests/test_lang_parity.py` 実行時点で全追加キーが参照済みとなり未使用キー assertion を回避する（`_ALLOWLIST` ハックは不要）。
- **懸念3（MEDIUM）再実行制御フローの欠落**: 反映済み。Task 2 に `_set_running_ui(running)` を追加し、実行中は「▶ 実行」「+ ファイル追加」「削除」を disabled・「バッチ中止」を enabled、中止/完了後は逆へ戻す。再実行時は `count_pending`（STATUS_PENDING）のみを処理対象とし STATUS_DONE は再送信しない。Task 3 に `test_rerun_skips_completed` を追加。
- **懸念4（MEDIUM）事前ページ数スキャンの UI ブロッキング**: 反映済み。D-03（集約コスト確認は総ページ数を要するため投入時に事前スキャン）を維持しつつ、`_scan_page_counts` で「読み込み中...」ステータス（`batch_scanning_msg`）表示 + ループ内 `update_idletasks()` により UI 応答性を確保する。遅延取得（OCR ループ直前で page_count 取得）案は D-03 の投入時集約コスト表示と行程が変わるため不採用とし、応答性メッセージ方式を採用した（rejected 代替の記録）。
- **懸念5（MEDIUM）`OCRDialog` インスタンスメソッド流用方式の曖昧さ**: 反映済み。`_confirm_cost`/`_estimate_cost`/`_check_cloud_api_key`/`_is_cloud_provider` を `BatchOCRDialog` へ**同一シグネチャ・同一挙動のコピペ移植**として自前実装することを Task 1 action・acceptance・Artifacts に明記。`ocr_dialog.py` は変更せず継承/cross-import もしない。
- **懸念6（LOW・04-01 由来）STATUS_ERROR 進捗集計**: 反映済み。BatchState を `BatchState(total_files=count_pending(entries))`（STATUS_ERROR 除外）で構築する連携を Task 2 に明記（04-01 で `count_pending` を新設）。
