  <name>挿入 Undo/Redo の内容同一性検証 + 全 op 往復の安全網</name>
  <files>tests/test_pdf_ops.py</files>
  <behavior>
    挿入 Undo/Redo（D-07）:
    - do: 3 ページ doc の指定位置に N ページを挿入 → `len(doc) == 3 + N`。
    - undo: `len(doc) == 3` に戻る、かつ残った元 3 ページの digest が挿入前と一致（内容同一性）。
    - redo: `len(doc) == 3 + N` に戻る、かつ挿入ページの内容が復元（挿入ページ digest 一致）。

    全 op 往復（安全網）:
    - rotate: 90度回転 → undo で 0 → redo で 90。
    - delete: 1 ページ削除 → undo で復元（digest 一致）→ redo で再削除。
    - move/duplicate/merge/bulk_move/bulk_crop/merge_resize: do→undo で元状態 → redo で操作後状態。各々 len と代表 digest または属性（rotation/cropbox/順序）で検証。

    全ケースで pdf_bytes キーに依存しないこと（state に "pdf_bytes" が現れない、または実 _restore_state が dict を返す対称機構を経由する）。
  </behavior>
  <implementation>
    RED→GREEN を意識して、まず失敗するテストを書いてから 01-01/01-02 実装に対して GREEN を確認する（本プランは 01-01/01-02 完了後の wave 3 なので、テストは初回実行で GREEN になる想定だが、対称機構を直接叩く形で書くこと）。

    実装方針（read_first を踏まえ executor 裁量）:
    - 既存テストは fitz API を直接叩く「同等ロジック」方式（TestUndoRedoLogic 参照）と、`PDFEditorApp` のメソッドを `__get__` で借りる方式（TestCheckSplitOverwrite 参照）の 2 系統がある。本テストは Undo/Redo 機構そのもの（`_save_undo`/`_undo`/`_redo`/`_restore_state`）を検証対象とするため、可能なら実メソッドを最小フェイク app（`self.doc`/`self.current_page`/`self.selected_pages`/`self._undo_stack`/`self._redo_stack`/`MAX_UNDO`/`_invalidate_thumb_cache`/`_preview_gen`/`_thumb_gen`/`_refresh_all`/`_set_status`/`_t` をスタブ）に bind して呼ぶ。Tkinter 非依存メソッドのみを叩き、UI 系（`_refresh_all` 等）は no-op スタブにする。
    - ページ digest は `page.get_text()` の文字列ハッシュ、または `tmp.tobytes()` の bytes ハッシュを使う（conftest の sample は "Page N"/"FileN PageM" テキストを持つため text ベースが簡潔）。multi_pdf_files フィクスチャ（1/2/3 ページの別内容 PDF）を挿入ソースに使う。
    - merge_resize は A4×2 → A3 等の最小ケースで len と挿入ページ rect で検証。
    - 既存 `test_fallback_pdf_bytes_format` は D-05 で撤廃された pdf_bytes 分岐に対応するため、削除または「Undo/Redo state に pdf_bytes キーが生成されない」検証へ置換し、SUMMARY に判断を記載する。
    - fenced code block でのフル実装は read_first の既存テスト記法に合わせて executor が記述する。
  </implementation>
</feature>
