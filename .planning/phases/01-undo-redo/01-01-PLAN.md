<objective>
Undo/Redo を「順方向も巻き戻しも同じ op 別デルタ機構を使う対称設計」へ作り替え、`doc.tobytes()` によるフルシリアライズを全パスから排除する（BUG-02 / D-01・D-04・D-05）。あわせて、巻き戻し時にページ増減系（insert/merge）が削除するページの bytes をキャプチャして Redo 用デルタへ保持することで、ページ挿入 Undo が正しく元へ戻り、Redo で挿入ページが内容ごと復元されるようにする（BUG-01 / D-04）。

Purpose: 成功基準#1（挿入 Undo が正確に戻る）と成功基準#3（大きな PDF で Undo してもフリーズしない）の土台を、全 op 対称化という単一の機構で同時に満たす。`pdf_bytes` を一切持たない状態がゴール。
Output: 対称デルタ化された `_undo`/`_redo`/`_restore_state`（file_ops.py）、insert/merge の redo 用 bytes キャプチャ、merge_resize の op 別デルタ化と `_do_insert` の挿入数書き戻し（page_ops.py）。

## 明示的に遵守する不採用制約（executor は以下のアプローチを導入しないこと）

- **D-02:** 背景スレッドでの `doc.tobytes()` は採用しない。fitz の `Document` はスレッド間で共有できず、データ競合・クラッシュのリスクが高い。本プランの対称デルタ化は UI メインスレッド上で完結させ、Undo/Redo 用に別スレッドでの全体シリアライズを一切導入しない。
- **D-03:** 元ファイルからの順方向全再適用方式（操作列を最初から再生して状態を再構築する方式）も採用しない。操作数増加で遅くなりうるため。本プランは「いま実行する逆操作のデルタ」のみを積む対称デルタ方式に限定する。

> 上記2点は選択した逆操作デルタ方式（D-01/D-04/D-05）が構造的に満たす制約であり、検証時はこれらの不採用アプローチのコードが新規に混入していないことを併せて確認する。
</objective>

**pagefolio/file_ops.py（変更）**
- `_undo()` — `_redo_stack` へ「pdf_bytes 辞書」ではなく `_restore_state` が返す逆操作 op 別 state を push するよう書き換え。

- `_restore_state(state)` — `pdf_bytes` 分岐を撤廃（D-05）。各 op の逆操作を適用しつつ「逆方向に戻すための新しい op 別 state（=逆デルタ）」を構築して返す。insert/merge の巻き戻し（ページ削除）時に、削除するページの bytes をキャプチャして返却デルタの `data` に格納する。

- `_save_undo("insert", ...)` の `state["data"] = [insert_at, 0]` スキーマは維持（page_ops 側で総数を書き戻す）。

**pagefolio/page_ops.py（変更）**
- `_do_insert()` — `self._undo_stack[-1]["data"][1] = total` の挿入総数書き戻しを維持・確認（BUG-01）。deque 化前提でも `[-1]` 直近エントリ参照が成立することを保証。
- `_do_merge_resize()` — 現状の `pdf_bytes` スナップショット push（`self._undo_stack.append({"pdf_bytes": ...})`）を撤廃し、op 別デルタ（例 `_save_undo("merge_resize", targets=..., ...)` または既存 op の組み合わせ）へ置換（D-05）。挿入された結合ページの bytes と削除された元ページの bytes をキャプチャして対称復元できる state を構築。

**逆デルタ（redo エントリ）スキーマ**
- Undo エントリと同一スキーマ（`op` / `current_page` / `selected_pages` / `data`）に揃える。`pdf_bytes` キーは一切使わない。
- insert の redo エントリ: 削除したページ群を `(page_index, page_bytes)` の昇順リストとして `data` に保持し、redo 時に `delete` 相当の再挿入で復元する設計（具体構造は executor 裁量、D-04）。
