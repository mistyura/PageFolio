## v1.6.0 — 2026-06-18: 設定/UI 改善（OCR パラメータ一元化・スライダー配置）

> マイルストーン v1.6.0（GSD Phase 1）。OCR パラメータ設定の二重入力を解消し、サムネイルサイズ変更スライダーを左ペイン縮小時も操作できる配置へ改善。
> 要件・ロードマップ詳細: `.planning/REQUIREMENTS.md` / `.planning/ROADMAP.md`

### UI 改善 — OCR パラメータの一元化（V16-UI-01 / Phase 1 Plan 01）

- **OCR 抽出画面の数値パラメータを読み取り専用化**: `OCRDialog` の数値パラメータ（scale / timeout / max_tokens / temperature）を `state="readonly"` + `fg=C["TEXT_SUB"]` でグレーアウト表示し、`model_combo`・モデル取得ボタンを `disabled` 化。「今どの設定で OCR が走るか」が見える状態は維持しつつ、編集導線を「⚙ LLM 設定…」（`LLMConfigDialog`）へ一元化（D-01／D-02）。
- **設定変更の即時反映**: 数値同期を `_sync_param_vars_from_settings` に切り出し、`_apply_llm_settings` のプロバイダ分岐外（全プロバイダ共通箇所）から呼び出すことで、LM Studio だけでなく Claude / Gemini でも LLM 設定適用直後に読み取り専用表示へ即時反映（D-03）。
- 実行ごとに変える実行時オプション（プロンプトプリセット・埋め込みテキスト無視・セッション API キー）は二重化の対象外として編集可能のまま OCR 画面に残置（D-06）。

### UI 改善 — サムネイルスライダーの配置（V16-UI-02 / Phase 1 Plan 02）

- **サムネイルサイズ変更スライダーを独立全幅行へ移設**: 従来は全選択/解除ボタンと同一行（`sel_frame` 内 `side="right"`）にあり、左ペインを縮小するとボタンと幅を奪い合って細くなり操作できなくなっていた。ボタン行の直後・サムネイル canvas の前に専用の全幅独立行（`zoom_frame`）を新設し、スライダーを `fill="x"` で全幅配置（D-07／D-08）。
- スライダーの挙動（範囲 0.5〜2.5・`thumb_zoom_var`・`<ButtonRelease-1>` → `_on_thumb_zoom_release` の設定保存と再描画）は不変。配置のみの変更で `viewer.py` / `settings.py` は未変更（D-09）。

### 詳細

#### 機能追加: ページ→画像変換（1ページ1ファイル）

- ページを画像ファイル（1ページ1ファイル）として保存する機能。AI（LLM）にテキストを
  読み取らせる用途を想定し、出力サイズは DPI ではなく**長辺ピクセル数**で指定する
  （プリセット: 1024 / 1568 / 2048 / 3072。Claude≈1568 / Gemini≈3072 が目安）。
- 対象ページは「全ページ / 選択ページ / 範囲指定（分割と同じ書式）」の 3 モード。
- 形式は PNG（既定・文字がくっきり）と JPEG（品質 10〜100・サイズ小）。
- ファイル名は分割機能と同じ `{元ファイル名}_p{ゼロ詰めページ番号}.{拡張子}` 形式で、
  保存先の同名ファイルは既存の上書き確認（`_check_split_overwrite`）を再利用。
- レンダリングは `page.rect`（CropBox・回転反映後）基準のため、アプリ内で編集した
  トリミング・回転の結果がそのまま画像に反映される。
- `fitz` はメインスレッド限定の制約に従い、変換はメインスレッドで 1 ページずつ実行し、
  ステータスバーに進捗（`画像変換中… n/total`）を表示する。

#### 機能追加: OCR の続きから再実行（リラン / リスタート）

- **背景**: クラウド OCR（Gemini / gemma 系等）でタイムアウト等の致命的エラーが発生すると
  実行全体が中断され、成功済みページの結果があっても最初からやり直す必要があった
  （実例: 18 ページ中 12 ページ成功後のタイムアウトで全結果が再取得対象に）。
- **リラン（全体再実行）**: 実行終了後（完了 / エラー / キャンセル）に「▶ 読み取り実行」を
  自動で再有効化。従来必要だった「クリア」を経由せずに全体を再実行できる
  （押下時に前回の結果・エラー・スキップ状態を破棄して開始する）。
- **リスタート（続きから再実行）**: 「⏯ 続きから再実行」ボタンを新設。
  成功済みページの結果を保持したまま、エラー / 未処理ページのみ再実行する。
  完了時は全ページ分をページ順に統合して表示する。部分的な結果が残っている場合のみ
  有効化され、結果欄に「未処理 {n}ページ（p.{first}〜）」の再開案内を追記する。
- **コスト見積の整合**: クラウド実行前のコスト確認は、再開時には未処理ページ数のみで
  見積もる（`_confirm_cost(page_count)`）。
- **進捗表示**: 再開実行では進捗バー・カウントを「今回の実行対象ページ数」基準で表示する
  （`_run_pages` / `_skip_base` による実行単位の進捗管理）。

#### 機能追加: サーキットブレーカーとエラー件数つき完了表示

- **背景**: 5xx/429 はページ単位リトライ（最大3回）後にエラー記録して続行する設計のため、
  サーバ側が完全に落ちていると全ページ × リトライ待機を無駄に消化する。また
  エラーページがあっても完了表示が「完了: n/total」のままで失敗に気づきにくかった。
- **サーキットブレーカー**: リトライ上限到達が **3ページ連続**したら実行を中断する
  （`kind="circuit_breaker"` の致命的エラー扱い）。途中で1ページでも成功すれば
  連続カウンタはリセットされる。中断後も成功済み結果は保持され、結果欄に
  「{n}ページ連続でエラー → 時間を置いて『続きから再実行』」の案内を表示する。
- **完了表示の改善**: エラーページが残ったまま完了した場合は
  「⚠ 完了: {count}/{total} ページ成功・エラー {err}ページ」を **警告色（WARNING）** で
  表示する（全ページ成功時は従来どおり）。次回実行開始時・クリア時に通常色へ戻す。

#### UI 改善: OCR ダイアログヘッダーの整理

- プロバイダ表示行を「左寄せ: OCR プロバイダ名 + モデル名 / 右寄せ: ⚙ LLM 設定ボタン」の
  配置に変更（従来はボタンが左側・モデル名が右端）。
- ボタン行が 7 ボタンになったため、ダイアログの最小幅を拡大（1060→1150px 基準）。

#### バグ修正: 「縮小して保存」で元ファイルへ上書き保存できない

- **症状**: 「縮小して保存」で現在開いているファイル自身を保存先に指定すると
  エラーになり上書き保存できない（元ファイルがホールドされ permission エラー）。
- **原因**: `fitz.Document` が元ファイルのハンドルを保持しているため、PyMuPDF は
  同一パスへの非インクリメンタル `save()` を拒否し、Windows ではファイルロックにより
  `os.replace` も PermissionError になる。
- **対応**: 保存先が開いているファイル自身の場合（`_is_current_file` で判定）、
  ① `doc.tobytes()` でメモリ上へシリアライズ → ② `doc.close()` でハンドル解放 →
  ③ tmp ファイル経由で `os.replace` → ④ 新ファイルを開き直す手順
  （`_overwrite_current_file`）で上書きする。書き込み失敗時はメモリ上の bytes から
  doc を復元して操作を継続できる。`_save_file` の非インクリメンタル保存
  フォールバック（同じく Windows でロック起因の失敗があり得る）も同方式に変更した。

### 詳細

- **症状**: OCR テキスト抽出でタイムアウト（致命的エラー）発生後、タイムアウト値を変更して
  「クリア」→「読み取り実行」しても処理がすぐに終了し、前回のエラーがそのまま表示される。
  ダイアログを閉じて開き直すと正常に読み取りが始まる。
- **原因**: `_finish_error` で記録された `_fatal_msg` / `_fatal_kind` と完了カウンタ
  `_done_count` が `_clear_text` でリセットされていなかった。再実行時に全ワーカーが
  `has_fatal=True` と判定して API 呼び出しをスキップし、最終ワーカーが残留した旧エラーで
  `_finish_error` を即座に呼んでいた（ダイアログを開き直すと新インスタンスになるため再現しない）。
- **対応**: `_clear_text` に加えて `_on_run`（実行開始時）でも `_fatal_msg` / `_fatal_kind` /
  `_done_count` / `_done` を初期化し、クリアを経由しない再実行経路でも残留状態を持ち込まないようにした。

#### H-7: Gemini API 経由の gemma モデル選択時に 400 エラー

- **症状**: Gemini プロバイダでモデル一覧から gemma 系モデル（例: Gemma 4 26B）を選択して
  読み取り実行すると HTTP 400 エラーが発生する。
- **原因**: `_build_payload` が「モデル名に `pro` を含まない場合は常に `thinkingConfig` を送る」
  実装だったため、thinkingConfig 非対応の gemma 系モデルで 400 INVALID_ARGUMENT が返っていた。
  `list_models` は generateContent 対応モデルをすべて返すため gemma を選択できてしまう。
- **対応**: thinkingConfig の送信条件を `model.startswith("gemini") and "pro" not in model`
  に変更し、gemini の non-pro（flash 等）のみに限定した。

#### 改善: 429/5xx エラーメッセージの分離

- **背景**: 旧実装は HTTP 429 と 5xx を同一文言「レート制限またはサーバエラー」で表示しており、
  HTTP 500（Google 側内部エラー）をレート制限と誤認させていた。
- **対応**: `OCRRetryableError` に `code` 属性を追加し、429 は「HTTP 429: レート制限」、
  5xx は「HTTP 5xx: サーバエラー」と表示を分離。リトライ待機中の進捗表示も
  「レート制限のため待機中」/「サーバエラーのためリトライ中」に分岐する。
  code 不明（プラグイン由来等）はサーバエラー側へ倒す。
- **確認方針**: 429 は Tier 1 環境では意図的に再現しにくいため、実機確認は 500 側のみ実施し、
  429 側は単体テスト（`TestRetryableErrorMessageSplit` / `TestRetryWaitKey`）で担保する。

#### 機能追加: 現在選択モデルの表示

- **背景**: クラウドプロバイダ（Claude / Gemini）選択時は LM Studio のモデル欄が非表示になり、
  どのモデルで OCR されるのかダイアログ上で確認できなかった。
- **対応**: プロバイダ表示行に「モデル: <モデル名>」を併記。claude / gemini は settings の
  各モデルキー、lmstudio は Combobox のライブ値（trace で即時反映）、tesseract は非表示。
  LLM 設定ダイアログでの変更時も `_refresh_provider_dependent_ui` 経由で更新される。
- **補足（実機切り分け結果）**: Gemini API 経由の gemma 系モデルは解像度 1.5 でも HTTP 500 が
  発生し、gemini-2.5-flash は解像度 4.0 でも成功することを確認。gemma の 500 はアプリ側では
  解消できないため README に注意書きを追加した。

#### 機能追加: 埋め込みテキスト無視オプション

- **背景**: `has_embedded_text` は非空白文字数のしきい値のみで判定するため、スキャナ付属 OCR
  などが埋め込んだ低品質なテキストレイヤーも無条件にスキップされ、ユーザー側に回避手段がなかった。
- **対応**: OCR ダイアログの詳細設定の下に「埋め込みテキストを無視して OCR を実行（スキップしない）」
  チェックボックスを追加。ON にすると `_render_next_page` の埋め込みテキスト判定を行わず、
  全ページを Vision OCR キューへ積む。
- **設計判断**: クラウド課金に直結するスイッチのため設定ファイルへは永続化せず、ダイアログを
  開くたびに OFF へ戻す。コスト確認ダイアログの見積もりは元々全ページ数ベースのため変更不要。
