## v1.7.4 — 2026-07-09: プロンプト外部ファイル読込・Markdown 表示のプリセット一本化・モデル取得タイムアウト見直し

### 1. カスタム/サマリプロンプトの外部 md ファイル読込（V174-2）

**背景**: 経理書類 OCR のような業務用途ではカスタムプロンプト・サマリプロンプトが数十行規模に巨大化し、LLM 設定ダイアログの小さな入力欄（3 行の Text）での編集・管理が現実的でなくなる。外部エディタで管理できるファイル方式が求められた。

**対応内容**:
- 実行ファイル（開発時はプロジェクトルート）と同じ階層に置いた `ocr_custom_prompt.md` / `ocr_summary_prompt.md` と LLM 設定の入力欄を**双方向連動**させる「ファイル連動モード」を新設（ファイル名は `constants.py` の `CUSTOM_PROMPT_FILE` / `SUMMARY_PROMPT_FILE`）。
  - **開いたとき**: ファイルが存在して非空なら、その内容を LLM 設定ダイアログのプロンプト入力欄へ反映する（画面でファイルの中身を確認・編集できる）。
  - **適用したとき**: ファイルが存在する場合のみ、入力欄の内容をファイルへ書き戻す（`save_prompt_file`・settings にも同期保存）。ファイルを使わないユーザーには md を新規作成しない（従来どおり settings のみで完結・後方互換）。
  - **実行したとき**: OCR 実行・サマリ実行のたびにファイルを毎回再読込し（`_on_run` / `_on_summary`）、外部エディタでの編集もアプリ再起動なしで次回実行へ反映される。
- 読込層は `settings.py` の `load_prompt_file` / `prompt_file_exists` / `save_prompt_file` / `load_custom_prompt` / `load_summary_prompt`。エンコーディングは UTF-8（読込は Windows エディタの BOM 付き `utf-8-sig` も許容）。無効時（不在/空/読込失敗）は設定欄の値へフォールバックする。
- 配置基準ディレクトリの解決は `_get_settings_path` から抽出した `_get_base_dir()` へ一元化（`pagefolio_settings.json` と同じ場所）。
- LLM 設定ダイアログでは、ファイル検出時のみ各プロンプト欄の直下に「📄 {file} と連動中 — 適用時にこの欄の内容をファイルへ保存します」の注記を WARNING 色で表示する（`_add_prompt_file_notice`・空ファイルでも連動対象のため存在のみで判定）。

### 2. Markdown 整形表示のプリセット一本化（V174-3）

**背景**: 巨大なカスタム/サマリプロンプトは出力形式（Markdown テーブル等）もプロンプト側で指示するが、カスタムプロンプト使用時は OCR ダイアログのプリセット選択（通常テキスト/表形式/Markdown）が実プロンプトへ反映されない（`resolve_ocr_prompt` でカスタムが最優先）。当初は LLM 設定に「Markdown整形で表示」チェックボックス 2 個（`ocr_custom_prompt_markdown` / `ocr_summary_markdown`）を新設して個別指定できるようにしたが、OCR ダイアログのプリセットと Markdown 指定が 2 箇所に重複するため、ユーザー判断でプリセットへ一本化した。

**対応内容**:
- 整形描画の判定は **OCR ダイアログのプリセット「Markdown」に一本化**。カスタム/サマリプロンプト使用時も `preset == "markdown"` で整形表示が決まる（`_render_results_ordered` / `_on_summary_done`）。コピー/保存が raw 維持である点は不変。
- カスタムプロンプト使用中は、プリセットが実プロンプトへ反映されず「表示形式の選択」としてのみ働くため、OCR ダイアログのプリセット横に「（カスタムプロンプト使用中 — プリセットは表示形式にのみ適用）」の注記を表示する（`_update_preset_note`・LLM 設定適用時と外部 md 再読込時に再評価）。
- チェックボックス 2 個と設定キー `ocr_custom_prompt_markdown` / `ocr_summary_markdown`、純関数 `resolve_render_markdown` は撤去（同一バージョン内の実装のため互換影響なし）。
- `lang.py` に `ocr_custom_prompt_label/hint`・`ocr_preset_custom_note` を ja/en 両方へ追加（キー数パリティ維持）。

### 3. クラウド LLM のモデル一覧取得タイムアウト見直し（V174）

**症状**: LLM 設定ダイアログの「モデル取得」で、モデル一覧取得のタイムアウトが全プロバイダ一律 10 秒にハードコードされており、クラウド LLM（特に RunPod Serverless の初回起動＝コールドスタート。ワーカー起動 + モデルロードで数十秒〜数分かかる）では `timed out after 10s` で取得に失敗するケースがあった。

**対応内容**:
- `OCRProvider` にクラス属性 `model_list_timeout`（既定 10 秒）を新設し、全プロバイダの `list_models`（Claude は `_fetch_models_page`）のハードコード 10 秒を置換。ローカル（LM Studio / Ollama）は即応するため 10 秒のまま、Claude / Gemini は 30 秒、RunPod はコールドスタートを見込み 90 秒へ引き上げ。
- タイムアウト延長に伴い、クラウド 3 プロバイダ（Claude / Gemini / RunPod）のモデル取得を共有ヘルパー `_fetch_models_async`（ワーカースレッド + `after(0)` でメインスレッドへ結果投函・二重実行ガード・ダイアログ破棄後は結果破棄）によるバックグラウンド実行へ変更し、取得待ちで UI がフリーズしないようにした。エラー時の静的推奨リストフォールバック（D-08）等の既存挙動はコールバック側で維持。
- RunPod の取得中ステータス文言に「初回起動時は 1 分程度かかることがあります」の注記を追加（ja/en）。

### 4. OCR ダイアログ右ペインの縦スクロール対応（V174-4）

**背景**: v1.7.3 でボタン群を右ペイン（縦積み・非スクロールの単純 Frame）へ移動した際、「将来ボタンが増えて縦に長くなる場合はメイン画面同様のスクロール対応を検討」と申し送りされていた。フォントサイズ最大（16）ではセクション群の要求高が約 330px となり、最小ウィンドウ高（620px）でも他のヘッダー行が伸びる分、右ペインの可視高（約 208px）を超えて下側のボタンが見えなくなる。

**対応内容**:
- 右ペインのセクション群（「▶ 実行」「📋 結果」）を縦スクロール可能な Canvas + Scrollbar 構成へ変更（`ui_builder._build_tools_scrollable` と同型: inner Frame を `create_window` で載せ、`<Configure>` で scrollregion / 幅同期、マウスホイールを子ウィジェットへ再帰バインド）。Canvas 幅は内容の要求幅（約 229px）に追随させ、従来のペイン幅を維持。
- 「✕ 閉じる」はスクロール領域外（右ペイン最下部固定）に据え置き、どの高さでも常時可視。
- Xvfb 実測: 既定フォント（10）・既定サイズ（1150x680）ではスクロール不要（内容 310px < 可視 349px）。最小サイズ（960x620）ではフォント 10/16 ともスクロールで最下部ボタン（クリア）まで到達可能・「✕ 閉じる」常時可視を確認。**アイコンのみの短縮ラベル化は不要と判断**（幅は最長ラベル基準 229px で問題なく、高さはスクロールで吸収できるため）。

**テスト**: 外部プロンプトファイル読込・書き戻し（優先順位/フォールバック/BOM/空ファイル/round-trip・12 件）、`_apply` のファイル書き戻し（存在時のみ・2 件）、`_on_summary_done` のプリセット判定（カスタムサマリプロンプト使用時含む・1 件）、`model_list_timeout` のプロバイダ別値・urlopen への伝播（6 件）を追加。テスト 880 件グリーン。

## v1.7.3 — 2026-07-08: OCR ダイアログのボタン群を右ペインへ再配置

**症状**: OCR テキスト抽出ダイアログ（`OCRDialog`）を初期状態で開いたとき、下部に横一列で並ぶ操作ボタン群のうち「⏯ 続きから再実行」などが画面幅に収まらず隠れてしまうことがあった。

**原因**: ボタン 8 個（コピー/保存/サマリ/クリア + 再開/実行/キャンセル/閉じる）を 1 つの `btn_row` に `pack(side="left")` / `pack(side="right")` で横並びに配置していたため、ウィンドウ幅がボタン合計幅（フォントサイズ・言語・OS のボタン余白に依存）を下回ると、pack 順で後発のボタンから順にクリップされる構造だった。`_center` の初期幅計算（`max(1150, fs * 90)`）では環境差を吸収しきれない。

**対応内容**:
- ボタン群を下部の横一列から、**結果テキスト右側の縦型右ペイン**へ移動（メイン画面右パネル `_build_tools` と同じ「セクション見出し（`BG_CARD` + `WARNING` 色）+ 全幅ボタン」構成）。
  - 「▶ 実行」セクション: 読み取り実行（Accent）/ 続きから再実行 / キャンセル（Danger）
  - 「📋 結果」セクション: クリップボードにコピー / テキストファイルに保存 / サマリ作成 / クリア
  - 「✕ 閉じる」は右ペイン最下部に固定（`pack(side="bottom")`・誤操作分離）
- 縦積みの全幅ボタンにしたことで、ウィンドウ幅がボタン表示に影響しない構造となり「幅不足でボタンが隠れる」問題を構造的に排除。最小サイズ（960x620）でも全ボタンが可視であることを Xvfb 上のスモークテストで確認。
- ボタンの属性名（`run_btn` / `resume_btn` / `cancel_btn` / `copy_btn` / `save_btn` / `summary_btn` / `clear_btn` / `close_btn`）と有効/無効の状態遷移ロジックは一切変更なし（レイアウトのみの変更）。
- `lang.py` にセクション見出しキー `ocr_sec_run` / `ocr_sec_result` を ja/en 両方へ追加（キー数パリティ維持・`test_lang_parity.py` グリーン）。

## v1.7.2 — 2026-07-06: LLM 設定ダイアログのスクロール対応・リサイズ許可

**症状**: PC（画面解像度・フォントサイズ）によっては LLM 設定ダイアログ（`LLMConfigDialog`）の内容が画面の高さに収まらず、Apply/Cancel ボタンをはじめ下部の項目に到達できない不具合が報告された。

**原因**: `LLMConfigDialog` は `resizable(False, False)` でウィンドウサイズが固定され、かつダイアログの高さを常に本文の必要高さ（`winfo_reqheight()`）に合わせて再計算していた。画面の物理的な高さより本文が高くなるケース（低解像度ディスプレイ・大きめのフォントサイズ設定・多くのプロバイダ固有欄が展開された状態など）を考慮しておらず、画面からはみ出しても補う手段（スクロール）がなかった。

**対応内容**:
- 本文全体を `tk.Canvas` + `ttk.Scrollbar`（縦）によるスクロール可能領域（`self._body`）に構築し直した（`_build_scrollable_area`）。マウスホイール（Windows: `<MouseWheel>`、X11: `<Button-4>`/`<Button-5>`）でのスクロールにも対応。
- ウィンドウを `resizable(True, True)` にしてユーザーによる手動リサイズを許可（`minsize(420, 320)`）。
- ダイアログの高さは `_compute_dialog_height()` で「本文＋ボタン行の必要高さ」と「画面の高さ − 余白」を比較し、小さい方にクランプするよう変更（画面をはみ出す場合はスクロールバー経由でアクセス）。
- Apply/Cancel ボタン行（`self._btn_row`）はスクロール領域の外に置き、`pack(side="bottom")` で常に画面内の下部に固定表示されるようにした（pack のパック順を制御し、ボタン行を先に確保してから残りをスクロール領域に割り当てる構成）。
- プロバイダ/モデル切替時の高さ再計算（`_resize_to_fit`）も同じクランプ処理を使うように統一。手動リサイズ済みの場合は現在の幅を尊重する。
- 仮想環境（`venv`）内の主要な依存パッケージ（`PyMuPDF` 1.28.0, `Pillow` 12.3.0, `tkinterdnd2` 0.6.2, `pyinstaller` 6.21.0, `pytest` 9.1.1, `pytest-cov` 7.1.0, `ruff` 0.15.20）を最新バージョンにアップグレードし、`requirements.txt` の固定バージョンを更新。最新環境でのテストパスを確認した。

**追加修正（H-7）**: 上記対応後、ウィンドウを狭くリサイズすると Claude/Gemini/RunPod の APIキー入力欄横にある表示切替ボタン（👁）が潰れて見えなくなる不具合が判明。原因は `pack(side="left")` で Entry（`fill="x", expand=True`）をボタンより先にパックしていたため、幅が不足するとパック順で後発のボタン側が優先的に切り詰められていたこと。3 箇所の APIキー行すべてでボタンを `pack(side="right")` に変更し Entry より先にパックすることで、幅が足りない場合は Entry 側が縮み、ボタンは常に必要幅を保持するよう修正した。

## v1.7.0 — 2026-07-03: Undo/Redo メモリ最適化（ディスク退避）+ ストレステスト自動化

> 大規模 PDF 処理時のメモリ圧迫問題の根本解決。undo デルタのページ bytes を
> 一時ファイルへ退避しメモリから解放する機構を導入し、100 ページ超 PDF での
> Undo/Redo 連続実行のメモリ推移を検証する自動ストレステストを追加した。
> 併せて undo が実質 no-op だった 3 操作のバグと、透かし追加が
> ValueError で失敗していたバグを修正。

### 前提の整理（要件定義書との差分）

- 要件書の「操作のたびに doc.tobytes() で PDF 全体をメモリ保持」という前提は
  旧仕様（v1.3.0 で対称デルタ方式へ移行済み）。実際のメモリ消費源は
  delete / page_edit / insert 系 / merge 系 / merge_resize デルタが保持する
  **ページ単位 bytes**（最大 2 スタック × 20 世代）であり、本バージョンは
  これのディスク退避（要件書アプローチ案②）として実装した。

### 機能・非機能 — undo デルタのディスク退避（新規 `pagefolio/undo_store.py`）

- `UndoBlobStore`: ページ bytes が **64KiB 以上なら tempfile へ退避**
  （`FileBlob`・`pagefolio_undo_` 一時ディレクトリ）、未満はメモリ保持
  （`MemBlob`・小ページの I/O オーバーヘッド回避）。書き込み失敗時は
  MemBlob へフォールバックし undo 機能自体は維持。
- キャプチャは v1.6.5 で 1 点化した共有経路を `_capture_page_blob` へ差し替え、
  復元は `_blob_bytes`（Blob / 生 bytes 両対応の後方互換層）経由。
  `merge_resize` の merged_bytes / orig_pages も Blob 化。
- **Blob ライフサイクル管理**（deque 標準機能では溢れ時のフックが無いため
  FileOpsMixin 側にフックを実装。カスタム deque はテスト互換性のため不採用）:
  - `_push_evicting`: deque(maxlen) 溢れで evict される最古 state を解放してから append
  - `_clear_redo_stack`: redo クリア時に各 state を解放
  - `_undo`/`_redo`: 消費済み state を解放（逆デルタが同一 data を共有する
    insert_undo→insert_redo / merge_resize 系は identity 比較で除外）
  - `_clear_undo_stacks`: ファイルオープン/クローズ/アプリ終了時に両スタック解放 + purge
  - `atexit` でも purge（異常終了以外で一時ファイルは残らない）
- Windows 考慮: mkstemp の fd は書き込み後即 close・読み取りは都度 open
  （delete-on-close 系 API 不使用）。unlink 失敗（AV ロック等）は suppress し
  purge の rmtree(ignore_errors) と atexit の二段回収。

### バグ修正 — undo が実質 no-op だった 3 操作

- `insert_blank` / `watermark` / `page_numbers` op は `_save_undo`/`_restore_state`
  に分岐が無く、undo しても何も起こらなかった（データ無し state が積まれるのみ）。
- 白紙ページ挿入 → 既存 `insert` op を再利用（挿入後に件数 1 を確定）。
- 透かし追加・ページ番号印字 → `page_edit` op（適用前ページ bytes・v1.6.5 新設）へ置換。

### バグ修正 — 透かし追加の ValueError

- `insert_text(rotate=45)` は PyMuPDF では無効値（90 度単位のみ許容）で、
  **透かし追加自体が ValueError で失敗していた**（新規テストで発見）。
  ページ中心を pivot とした `morph=(pivot, fitz.Matrix(45))` による
  45 度回転へ修正。

### テスト — ストレステスト自動化（新規 `tests/test_undo_stress.py`）

- 120 ページ・1 ページ 64KiB 超（決定的ノイズ画像埋め込みで高解像度画像
  ベース PDF を模擬）のフィクスチャを生成し、以下を自動検証（実行 約 7 秒）:
  - **正当性**: 5 ページ削除 → undo → redo → undo × 25 サイクルでページ数・内容一致
  - **メモリ**: 30 サイクルの Python ヒープ増分 < 20MB（tracemalloc）+
    **Blob 不変条件**（ストア内ファイル数 ≤ ライブ state の Blob 数・
    クリア後にディレクトリ消滅）を主アサーションとしてリーク検出
  - **eviction**: MAX_UNDO+5 世代積んで evict 分の一時ファイルが物理削除
  - **page_edit 往復**: 黒塗り連続適用でも Blob が滞留しない
- 3 操作の undo 修正のラウンドトリップテスト（透かし文字列が undo で消える
  ことまで検証）を追加。新規 7 件（計 707 件）。

## v1.6.3 — 2026-07-01: OCR カスタムプロンプトの「1 回前反映」バグを修正

> OCR ダイアログを開いたまま「⚙ LLM 設定…」でカスタムプロンプトを変更 → 適用 → 続けて OCR を実行すると、
> 変更後ではなく **変更前（1 回前）のカスタムプロンプト**で OCR が実行されてしまう不具合を修正。

### バグ修正

**症状**: OCR ダイアログを開いたまま「⚙ LLM 設定…」ボタンからカスタムプロンプトを書き換えて適用しても、
その場で OCR を実行すると書き換え前のプロンプトが使われてしまう（ダイアログを一度閉じて再度開き直すと
正しいプロンプトが使われる）。

**原因**: `OCRDialog.__init__` はダイアログ生成時点の `app.settings["ocr_custom_prompt"]` を
`self.custom_prompt` にキャッシュするだけで、`_apply_llm_settings`（LLMConfigDialog の `on_apply`
コールバック）は `app.settings` の更新・永続化・`scale_var`/`timeout_var`/`max_tokens_var`/
`temperature_var` の再同期は行っていたが、`self.custom_prompt` の再同期が抜けていた。そのため
`_on_run` が参照する `resolve_ocr_prompt(preset, provider, self.custom_prompt)` は常に
ダイアログ生成時点のキャッシュ値を使い続けていた。

**対応内容**: `pagefolio/ocr_dialog.py` の `_apply_llm_settings` に
`self.custom_prompt = self.app.settings.get("ocr_custom_prompt", "")` を追加し、設定適用後に
最新値へ同期するようにした。

| ファイル | 変更内容 |
|----------|----------|
| `pagefolio/ocr_dialog.py` | `_apply_llm_settings` で `self.custom_prompt` を最新の `app.settings` 値へ同期する処理を追加 |
| `tests/test_provider_ui.py` | `_apply_llm_settings` 実行後に `custom_prompt` が最新値へ更新されることを検証する回帰テストを追加 |
| `pagefolio/constants.py` | `APP_VERSION` を `v1.6.3` に更新 |

### バグ修正 — 設定ダイアログの LLM 設定サブダイアログで「適用」が永続化されない

**症状**: メイン画面の「⚙ 設定」→「🔍 LLM 設定…」で開くサブダイアログでプロバイダやカスタム
プロンプトを変更して「適用」を押しても、設定ファイル（`pagefolio_settings.json`）に反映されない。
サブダイアログの「適用」を押した後、外側の設定ダイアログを「キャンセル」で閉じる、あるいは
アプリを終了すると、変更した LLM 設定が失われる。

**原因**: `SettingsDialog._open_llm_config` の `on_apply` コールバックは
`self.current_settings`（`SettingsDialog.__init__` で `dict(current_settings)` としてコピーした
ローカルな辞書）を更新するだけで `_save_settings` を呼んでいなかった。永続化は外側の
`SettingsDialog._apply`（自身の「適用」ボタン）が呼ばれ `callback(new_settings)` 経由で
`app._apply_settings` → `_save_settings` に到達して初めて行われる設計だったため、LLM 設定側の
「適用」だけでは実際には何も保存されず、ユーザーには「適用しても更新されない」ように見えていた
（`pagefolio/ocr_dialog.py` の `OCRDialog._apply_llm_settings` は即座に永続化するため、OCR 画面
経由で開いた場合とメイン設定画面経由で開いた場合とで挙動が食い違っていた）。

**対応内容**: `pagefolio/dialogs/settings.py` の `_open_llm_config` 内 `on_apply` で
`self.current_settings.update(llm_settings)` の直後に `pagefolio.settings._save_settings` を
呼び、LLM 設定サブダイアログの「適用」単体で即座に永続化されるようにした。

| ファイル | 変更内容 |
|----------|----------|
| `pagefolio/dialogs/settings.py` | `_open_llm_config` の `on_apply` で `_save_settings` を即時呼び出すよう修正 |
| `tests/test_provider_ui.py` | LLM 設定サブダイアログの「適用」が `_save_settings` を即座に呼ぶことを検証する回帰テストを追加 |

### バグ修正 — 設定ダイアログ／LLM 設定ダイアログの二重起動

**症状**: 上記「適用が永続化されない」不具合をユーザー側で調査したところ、実際には「⚙ 設定」ボタンを
連続クリックする等の操作で `SettingsDialog` が二重に生成されていたことが判明。二重起動すると
`current_settings`（各ダイアログが個別に持つコピー）が2つ独立に存在し、片方のダイアログで
LLM 設定を「適用」しても、もう片方のダイアログを先に「適用」/「キャンセル」で閉じると、そちらの
（古い）`current_settings` で上書き、または何もせず終了してしまい、結果的に変更が消えたように見える。
同様の二重起動は `SettingsDialog._open_llm_config`（🔍 LLM 設定…ボタン）・
`OCRDialog._open_llm_config`（⚙ LLM 設定…ボタン）でも起こり得る。

**原因**: `_open_settings` / `_open_llm_config`（両経路）のいずれも、ボタンが押されるたびに
無条件で新しいダイアログインスタンスを生成しており、既に同種のダイアログが開いているかどうかの
チェックが存在しなかった。

**対応内容**: いずれのメソッドにも「既存インスタンスが存在し `winfo_exists()` が真なら、新規生成せず
既存ウィンドウを `lift()`/`focus_force()` して前面へ出すだけで return する」ガードを追加した。

| ファイル | 変更内容 |
|----------|----------|
| `pagefolio/app.py` | `_open_settings` に `self._settings_dialog` を用いた二重起動ガードを追加 |
| `pagefolio/dialogs/settings.py` | `_open_llm_config` に `self._llm_config_dialog` を用いた二重起動ガードを追加 |
| `pagefolio/ocr_dialog.py` | `_open_llm_config` に `self._llm_config_dialog` を用いた二重起動ガードを追加 |
| `tests/test_provider_ui.py` | 3経路すべてで既存ダイアログが再利用されること、閉じた後は新規生成されることを検証する回帰テストを追加 |

### 詳細

#### 機能追加: ページ→画像変換（1ページ1ファイル）

- ページを画像ファイル（1ページ1ファイル）として保存する機能。AI（LLM）にテキストを
  読み取らせる用途を想定し、出力サイズは DPI ではなく**長辺ピクセル数**で指定する
  （プリセット: 1024 / 1568 / 2048 / 3072。Claude≈1568 / Gemini≈3072 が目安）。
- 対象ページは「全ページ / 選択ページ / 範囲指定（分割と同じ書式）」の 3 モード。
- 形式は PNG（既定・文字がくっきり）と JPEG（品質 10〜100・サイズ小）。
- ファイル名は分割機能と同じ `{元ファイル名}_p{ゼロ詰めページ番号}.{拡張子}` 形式で、
  保存先の同名ファイルは既存の上書き確認（`_check_split_overwrite`）を再利用。
- レンダリングは `page.rect`（CropBox・回転反映後）基準のため、アプリ内で編集した
  トリミング・回転の結果がそのまま画像に反映される。
- `fitz` はメインスレッド限定の制約に従い、変換はメインスレッドで 1 ページずつ実行し、
  ステータスバーに進捗（`画像変換中… n/total`）を表示する。

#### 機能追加: OCR の続きから再実行（リラン / リスタート）

- **背景**: クラウド OCR（Gemini / gemma 系等）でタイムアウト等の致命的エラーが発生すると
  実行全体が中断され、成功済みページの結果があっても最初からやり直す必要があった
  （実例: 18 ページ中 12 ページ成功後のタイムアウトで全結果が再取得対象に）。
- **リラン（全体再実行）**: 実行終了後（完了 / エラー / キャンセル）に「▶ 読み取り実行」を
  自動で再有効化。従来必要だった「クリア」を経由せずに全体を再実行できる
  （押下時に前回の結果・エラー・スキップ状態を破棄して開始する）。
- **リスタート（続きから再実行）**: 「⏯ 続きから再実行」ボタンを新設。
  成功済みページの結果を保持したまま、エラー / 未処理ページのみ再実行する。
  完了時は全ページ分をページ順に統合して表示する。部分的な結果が残っている場合のみ
  有効化され、結果欄に「未処理 {n}ページ（p.{first}〜）」の再開案内を追記する。
- **コスト見積の整合**: クラウド実行前のコスト確認は、再開時には未処理ページ数のみで
  見積もる（`_confirm_cost(page_count)`）。
- **進捗表示**: 再開実行では進捗バー・カウントを「今回の実行対象ページ数」基準で表示する
  （`_run_pages` / `_skip_base` による実行単位の進捗管理）。

#### 機能追加: サーキットブレーカーとエラー件数つき完了表示

- **背景**: 5xx/429 はページ単位リトライ（最大3回）後にエラー記録して続行する設計のため、
  サーバ側が完全に落ちていると全ページ × リトライ待機を無駄に消化する。また
  エラーページがあっても完了表示が「完了: n/total」のままで失敗に気づきにくかった。
- **サーキットブレーカー**: リトライ上限到達が **3ページ連続**したら実行を中断する
  （`kind="circuit_breaker"` の致命的エラー扱い）。途中で1ページでも成功すれば
  連続カウンタはリセットされる。中断後も成功済み結果は保持され、結果欄に
  「{n}ページ連続でエラー → 時間を置いて『続きから再実行』」の案内を表示する。
- **完了表示の改善**: エラーページが残ったまま完了した場合は
  「⚠ 完了: {count}/{total} ページ成功・エラー {err}ページ」を **警告色（WARNING）** で
  表示する（全ページ成功時は従来どおり）。次回実行開始時・クリア時に通常色へ戻す。

#### UI 改善: OCR ダイアログヘッダーの整理

- プロバイダ表示行を「左寄せ: OCR プロバイダ名 + モデル名 / 右寄せ: ⚙ LLM 設定ボタン」の
  配置に変更（従来はボタンが左側・モデル名が右端）。
- ボタン行が 7 ボタンになったため、ダイアログの最小幅を拡大（1060→1150px 基準）。

#### バグ修正: 「縮小して保存」で元ファイルへ上書き保存できない

- **症状**: 「縮小して保存」で現在開いているファイル自身を保存先に指定すると
  エラーになり上書き保存できない（元ファイルがホールドされ permission エラー）。
- **原因**: `fitz.Document` が元ファイルのハンドルを保持しているため、PyMuPDF は
  同一パスへの非インクリメンタル `save()` を拒否し、Windows ではファイルロックにより
  `os.replace` も PermissionError になる。
- **対応**: 保存先が開いているファイル自身の場合（`_is_current_file` で判定）、
  ① `doc.tobytes()` でメモリ上へシリアライズ → ② `doc.close()` でハンドル解放 →
  ③ tmp ファイル経由で `os.replace` → ④ 新ファイルを開き直す手順
  （`_overwrite_current_file`）で上書きする。書き込み失敗時はメモリ上の bytes から
  doc を復元して操作を継続できる。`_save_file` の非インクリメンタル保存
  フォールバック（同じく Windows でロック起因の失敗があり得る）も同方式に変更した。

### 詳細

- **症状**: OCR テキスト抽出でタイムアウト（致命的エラー）発生後、タイムアウト値を変更して
  「クリア」→「読み取り実行」しても処理がすぐに終了し、前回のエラーがそのまま表示される。
  ダイアログを閉じて開き直すと正常に読み取りが始まる。
- **原因**: `_finish_error` で記録された `_fatal_msg` / `_fatal_kind` と完了カウンタ
  `_done_count` が `_clear_text` でリセットされていなかった。再実行時に全ワーカーが
  `has_fatal=True` と判定して API 呼び出しをスキップし、最終ワーカーが残留した旧エラーで
  `_finish_error` を即座に呼んでいた（ダイアログを開き直すと新インスタンスになるため再現しない）。
- **対応**: `_clear_text` に加えて `_on_run`（実行開始時）でも `_fatal_msg` / `_fatal_kind` /
  `_done_count` / `_done` を初期化し、クリアを経由しない再実行経路でも残留状態を持ち込まないようにした。

#### H-7: Gemini API 経由の gemma モデル選択時に 400 エラー

- **症状**: Gemini プロバイダでモデル一覧から gemma 系モデル（例: Gemma 4 26B）を選択して
  読み取り実行すると HTTP 400 エラーが発生する。
- **原因**: `_build_payload` が「モデル名に `pro` を含まない場合は常に `thinkingConfig` を送る」
  実装だったため、thinkingConfig 非対応の gemma 系モデルで 400 INVALID_ARGUMENT が返っていた。
  `list_models` は generateContent 対応モデルをすべて返すため gemma を選択できてしまう。
- **対応**: thinkingConfig の送信条件を `model.startswith("gemini") and "pro" not in model`
  に変更し、gemini の non-pro（flash 等）のみに限定した。

#### 改善: 429/5xx エラーメッセージの分離

- **背景**: 旧実装は HTTP 429 と 5xx を同一文言「レート制限またはサーバエラー」で表示しており、
  HTTP 500（Google 側内部エラー）をレート制限と誤認させていた。
- **対応**: `OCRRetryableError` に `code` 属性を追加し、429 は「HTTP 429: レート制限」、
  5xx は「HTTP 5xx: サーバエラー」と表示を分離。リトライ待機中の進捗表示も
  「レート制限のため待機中」/「サーバエラーのためリトライ中」に分岐する。
  code 不明（プラグイン由来等）はサーバエラー側へ倒す。
- **確認方針**: 429 は Tier 1 環境では意図的に再現しにくいため、実機確認は 500 側のみ実施し、
  429 側は単体テスト（`TestRetryableErrorMessageSplit` / `TestRetryWaitKey`）で担保する。

#### 機能追加: 現在選択モデルの表示

- **背景**: クラウドプロバイダ（Claude / Gemini）選択時は LM Studio のモデル欄が非表示になり、
  どのモデルで OCR されるのかダイアログ上で確認できなかった。
- **対応**: プロバイダ表示行に「モデル: <モデル名>」を併記。claude / gemini は settings の
  各モデルキー、lmstudio は Combobox のライブ値（trace で即時反映）、tesseract は非表示。
  LLM 設定ダイアログでの変更時も `_refresh_provider_dependent_ui` 経由で更新される。
- **補足（実機切り分け結果）**: Gemini API 経由の gemma 系モデルは解像度 1.5 でも HTTP 500 が
  発生し、gemini-2.5-flash は解像度 4.0 でも成功することを確認。gemma の 500 はアプリ側では
  解消できないため README に注意書きを追加した。

#### 機能追加: 埋め込みテキスト無視オプション

- **背景**: `has_embedded_text` は非空白文字数のしきい値のみで判定するため、スキャナ付属 OCR
  などが埋め込んだ低品質なテキストレイヤーも無条件にスキップされ、ユーザー側に回避手段がなかった。
- **対応**: OCR ダイアログの詳細設定の下に「埋め込みテキストを無視して OCR を実行（スキップしない）」
  チェックボックスを追加。ON にすると `_render_next_page` の埋め込みテキスト判定を行わず、
  全ページを Vision OCR キューへ積む。
- **設計判断**: クラウド課金に直結するスイッチのため設定ファイルへは永続化せず、ダイアログを
  開くたびに OFF へ戻す。コスト確認ダイアログの見積もりは元々全ページ数ベースのため変更不要。
