Output: caplog ログ非出力テスト（settings + プロバイダ）・ソース実キースキャンテスト・ja/en LANG parity テスト（新規ファイル）・実 API 検証チェックリスト文書（D-08）・3 経路キー秘匿監査文書（D-10）。

H5 の堅牢化ロジック（クランプ・バックオフ・サーキットブレーカー）は test_ocr.py で網羅テスト済みのため**自動テストを重複追加しない**（D-09）。主成果物は実 API 検証チェックリスト文書。
</objective>

<locked_decisions>
本プランが従う確定事項（CONTEXT.md / RESEARCH.md 由来）:

- **D-07:** H5 検証は「実機相当の手順書中心」。実 API を常時叩く方式は採らない。クランプ/リトライはユニット/統合テストで自動担保済み、実 API 検証は手順書＋チェックリストでユーザーが任意実行・結果記録。
- **D-08:** 検証手順書はフェーズ内（`.planning/phases/03-ocr-a/`）にチェックリスト形式で 1 ファイル作成（手順・期待結果・結果記入欄）。GSD 検証フローと整合。開発履歴.md やテスト docstring には集約しない。
- **D-09:** 追加自動テストはギャップのみ。`clamp_retry_after`・サーキットブレーカー・指数バックオフ・Retry-After 優先・max_tokens クランプは既にテスト済み（test_ocr.py）のため**重複させない**。RESEARCH 結論: クランプは実在＆網羅テスト済み・429×トークン超過の追加統合ケースは不要。
- **D-10:** 証跡は「回帰テスト＋監査チェックリスト文書」の両立。設定ファイル経路は test_settings_keyguard.py で手厚いため、ギャップであるログ平文露出とソース埋め込みを回帰テスト化し、3 経路（設定/ソース/ログ）の監査チェックリスト文書を残す。
- **D-11:** ログ非出力は caplog ベースの回帰テストで担保。主要経路（最低限 `_save_settings` と全クラウドプロバイダ）でログにキー値が出ないことをアサート。caplog の関数/モジュール粒度は実装裁量。
- **D-12:** ソース埋め込み防止は自動スキャンテストで担保。キーらしいパターン（`sk-ant-`・`AIza`・長い base64）が無いことを pytest でアサート。テストフィクスチャ除外設計は裁量だが「実キーがソースに無いことを CI で再発防止できる」ことは必須。

**caplog テストの判定ルール（Pitfall 4・必達）:** `_save_settings` はキー**名**（`claude_api_key` 等）を `logger.error` で出す（settings.py 89-91 は仕様）。テストは「キー**値**（`sk-ant-...`）が `caplog.text` に出ないこと」のみをアサートする。「キー名が出ない」はアサートしない。

**ソーススキャンの誤検知回避（Pitfall 5・必達）:** スキャン対象を `pagefolio/` に限定し `tests/` を除外（`tests/` のダミーキー `sk-ant-secret-should-not-appear`・`AIza-gemini-secret` を実キーと誤検知しない）。閾値長は実キー長近辺（sk-ant は 100 字超、AIza は 39 字）に設定。
</locked_decisions>
