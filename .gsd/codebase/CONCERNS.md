# PageFolio — 懸念事項・リスクエリア (Concerns & Risks)

## 技術的負債 & 設計上のリスク

### 1. Tkinter のメインスレッド制約と非同期の競合
- **リスク内容**: プレビューやサムネイルのレンダリング処理は `root.after()` の連鎖により非同期にスケジュールされ、世代カウンタ (`_preview_gen` / `_thumb_gen`) で競合を回避しています。しかし、ユーザーが超高速でページ切り替えや操作を連打した際、Tkinter 内部の `PhotoImage` のガベージコレクションが追いつかず、一時的にメモリ消費量がスパイクする危険性があります。
- **対策状況**: 世代の異なる画像は描画されずに破棄されますが、画像オブジェクト自体の解放ラグは Tkinter の挙動に依存しています。

### 2. PyMuPDF (fitz) のスレッドセーフティ制約
- **リスク内容**: `fitz.Document` インスタンスはスレッドセーフではありません。同じ `doc` インスタンスを別スレッドから同時に操作すると、アプリケーションがセグメンテーションフォールトで即座にクラッシュします。
- **対策状況**: OCR 処理などのバックグラウンドタスクを実行する際は、メインスレッドであらかじめ対象ページを Base64 (PNG 文字列) に変換してからワーカーに渡す設計となっており、別スレッドが直接 `self.doc` を触ることはありません。将来の拡張でこのルールを破ると重大なバグに繋がります。

## 脆弱性・壊れやすい箇所 (Fragile Areas)

### 1. Windows 上での一時ファイル（Undo 用 Blob）のロック問題
- **脆弱な箇所**: `pagefolio/undo_store.py`
- **リスク内容**: Windows 環境では、アンチウイルスソフトやエクスプローラーのインデックスサービスなどの外部プロセスが、退避された一時ファイル (`page_*.pdf`) をロックすることがあります。このタイミングで Undo スタックが溢れて `os.unlink()` を呼び出すと、ファイルがロックされて削除に失敗し、解放漏れが発生します。
- **対策状況**: `contextlib.suppress(OSError)` で削除失敗時のクラッシュを防ぎ、アプリ終了時の `shutil.rmtree` によるディレクトリ一括削除と、`atexit` フックによる二段回収に処理を委ねています。ただし、アプリがタスクマネージャ等から「強制終了（タスクキル）」された場合、一時フォルダ（`pagefolio_undo_*`）が OS の `temp` ディレクトリ内に残留します。

### 2. 黒塗り・モザイク (Redaction) の不可逆性とズレのセキュリティリスク
- **脆弱な箇所**: `pagefolio/redact_ops.py`
- **リスク内容**: 黒塗りとモザイク処理は、元のテキストデータや画像を物理的に破壊して保存するセキュリティ上極めて重大な操作です。
- **リスク要因**: 画面上の表示座標（回転・ズーム・トリミングが適用された状態）から PDF の物理座標（未回転・未トリミング）への変換式 (`_derotate_rect`) が非常に複雑です。もしこの座標逆変換ロジックにバグがあると、**「黒塗りしたはずの領域の隣のテキストが露出したまま残り、無関係な場所が消える」**という機密漏洩リスクがあります。
- **対策状況**: `tests/test_pdf_ops.py` などで回転やトリミングを加えた状態の描画領域に対して適用テストを実施していますが、特に新しい PDF バージョンや複雑なレイアウトを持つファイルでの検証は継続的な注意が必要です。

## プラットフォーム依存

### 1. 印刷機能の OS 制約
- **脆弱な箇所**: `pagefolio/print_ops.py`
- **リスク内容**: `os.startfile(path, "print")` という Windows 独自の API に依存しているため、macOS や Linux 環境では機能せず、ボタンが無効化されます。クロスプラットフォーム化を進める際、印刷ハンドラの再設計が必要になります。
